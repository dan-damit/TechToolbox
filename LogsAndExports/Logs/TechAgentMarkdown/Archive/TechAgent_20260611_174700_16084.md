## Root Cause Analysis

Your hypothesis is correct. The user ddtest1 has no mailbox license in 365, and that causes the error. Here is the exact breakdown.

### What happens during Disable-User

When `Disable-User` processes EXO steps for ddtest1, it still attempts to call Exchange Online cmdlets (e.g., setting forwarding, granting FullAccess/SendAs). Because there is no mailbox, those EXO calls do not return structured objects (PSCustomObject or hashtable) as expected. Instead they return simple types:
- `$true` / `$false` (boolean)
- A string (error message or confirmation text)
- `$null`

### Where the error originates

In `Write-OffboardingSummary`, the code iterates over `$Results.Keys` and assigns each step to `$step`. Then it does this:

```powershell
if ($step -and $step.PSObject.Properties['Action']) { ... }
```

The member `.PSObject.Properties` exists only on objects that support reflection-PSCustomObject, PSObject wrappers, and hashtables (which have `.ContainsKey`). Simple booleans, strings, and `$null` do not expose `PSObject.Properties`. When one of those simple types is passed in as `$step`, PowerShell throws:

> The property 'Properties' cannot be found on this object. Verify that the property exists.

### Why it does not affect every step

Earlier steps in the transcript (e.g., AD disabling, local mailbox operations) likely returned PSCustomObject or hashtable objects, so those steps rendered fine. The EXO-specific steps for a user without a license are the ones that return non-object types and trigger the failure.

### Recommended fix in Write-OffboardingSummary

The function needs defensive type-checking before accessing `.PSObject.Properties`. A safe pattern:

```powershell
# Determine Action safely
$action = $null
if ($step -is [pscustomobject]) {
    if ($step.PSObject.Properties['Action']) { $action = $step.Action }
    elseif ($step.PSObject.Properties['Name'])  { $action = $step.Name }
}
elseif ($step -is [hashtable]) {
    if ($step.ContainsKey('Action'))       { $action = $step['Action'] }
    elseif ($step.ContainsKey('Name'))     { $action = $step['Name'] }
}

if (-not $action) { $action = $key }  # fallback to step key
```

For Success, use a similar check:

```powershell
$successVal = $null
$hasSuccessProp = $false

if ($step -is [pscustomobject]) {
    if ($step.PSObject.Properties['Success']) {
        $successVal = $step.Success
        $hasSuccessProp = $true
    }
}
elseif ($step -is [hashtable]) {
    if ($step.ContainsKey('Success')) {
        $successVal = $step['Success']
        $hasSuccessProp = $true
    }
}

# Handle simple types (boolean, string) that may have been returned
if (-not $hasSuccessProp) {
    if ($null -ne $step) {
        if ($step -is [bool]) {
            $successVal = if ($step) { "Success" } else { "Failed" }
            $hasSuccessProp = $true
        }
        elseif ($step -is [string]) {
            # Could be an error message or status string; treat as Unknown
            $hasSuccessProp = $false
        }
    }
}

# Apply Smart Defaults
if ($hasSuccessProp) { $success = $successVal }
elseif ($step.PSObject.Properties['Error']) { $success = "Failed" }
else { $success = "Unknown (No Status Data)" }
```

This prevents the error entirely and allows the summary to render even when a step returns a simple type. The `Success` line will likely show "Unknown (No Status Data)" for steps where the EXO call did not produce structured output-which is acceptable and accurate.
